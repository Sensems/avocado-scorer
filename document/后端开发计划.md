\## 一、 技术选型深度分析



基于 PRD 需求，我在你确定的 \*\*NestJS + PostgreSQL\*\* 基础上，建议引入 \*\*Redis\*\* 和 \*\*WebSocket\*\*。



\### 1. 为什么必须引入 Redis？



\* \*\*高频状态同步\*\*：记分过程是“视线不离牌桌”的高频操作。如果每次点击数字都直接读写 PostgreSQL，IO 压力大且响应慢。我们需要将“进行中”的房间数据（Room State）放在 Redis 中，实现\*\*毫秒级响应\*\*。

\* \*\*分布式锁\*\*：防止在“核爆清算”（1x3）或多人同时记分时出现竞态条件，导致分数对不上。



\### 2. 通信协议：WebSocket (Socket.io)



\* \*\*实时同步\*\*：PRD 要求“分数实时同步”。采用 WebSocket 可以在某位玩家（或房主代操）记分后，瞬间推送到房间内所有人的屏幕上，无需前端轮询。

\## 二、 数据库模型设计 (PostgreSQL)



我们需要兼顾“记分性能”和“战报追溯”的需求。



\### 1. 核心表结构设计



\#### \*\*Table: `users` (用户表)\*\*



存储微信用户信息及基础统计。

| 字段名 | 类型 | 说明 |

| :--- | :--- | :--- |

| `id` | `UUID` | 主键 |

| `openid` | `VARCHAR` | 微信唯一标识 (Index) |

| `nickname` | `VARCHAR` | 微信昵称 |

| `avatar\_url` | `TEXT` | 头像地址 |

| `created\_at` | `TIMESTAMP` | 注册时间 |



\#### \*\*Table: `rooms` (房间表)\*\*



| 字段名 | 类型 | 说明 |

| --- | --- | --- |

| `id` | `UUID` | 主键 |

| `room\_code` | `VARCHAR(6)` | 房间号 (高频查询索引) |

| `owner\_id` | `UUID` | 房主 ID |

| `status` | `ENUM` | `waiting` (待开局), `playing` (进行中), `finished` (已结算) |

| `config` | `JSONB` | 存储房间规则（如底分、快捷筹码配置等） |

| `created\_at` | `TIMESTAMP` | 创建时间 |



\#### \*\*Table: `room\_players` (房间玩家关联表)\*\*



处理真实玩家与“虚拟替身”的逻辑。

| 字段名 | 类型 | 说明 |

| :--- | :--- | :--- |

| `id` | `UUID` | 主键 |

| `room\_id` | `UUID` | 关联房间 |

| `user\_id` | `UUID` | 关联用户（虚拟位则为 NULL） |

| `is\_virtual` | `BOOLEAN` | 是否为虚拟替身 |

| `alias` | `VARCHAR` | 玩家代号（虚拟位显示名） |

| `position` | `INT2` | 2x2 宫格位置 (0-3) |

| `initial\_score` | `INT` | 初始分数 (默认 0) |

| `current\_score` | `INT` | 当前总分（实时冗余，提升查询效率） |



\#### \*\*Table: `score\_logs` (记分流水表 - 核心堆栈)\*\*



用于支撑“撤销”功能（Undo Stack）和战报引擎。

| 字段名 | 类型 | 说明 |

| :--- | :--- | :--- |

| `id` | `BIGINT` | 主键（自增，保证顺序） |

| `room\_id` | `UUID` | 关联房间 |

| `operator\_id` | `UUID` | 操作者 ID |

| `from\_player\_id` | `UUID` | 输家 ID |

| `to\_player\_id` | `UUID` | 赢家 ID |

| `amount` | `INT` | 变动分值 |

| `type` | `ENUM` | `normal` (1对1), `all\_in` (1对多), `undo` (撤销回滚) |

| `is\_revoked` | `BOOLEAN` | 是否已被撤销 |

| `created\_at` | `TIMESTAMP` | 操作时间 |

\## 三、 详细后端开发计划



我们将开发周期分为四个核心阶段（Sprint）。



\### \*\*第一阶段：基座搭建与房间系统 (Day 1-3)\*\*



\* \*\*Step 1: NestJS 项目初始化\*\*

\* 配置 TypeORM 与 PostgreSQL 连接。

\* 集成 Swagger 自动生成 API 文档。





\* \*\*Step 2: 鉴权与虚拟位逻辑\*\*

\* 实现微信小程序 `code2Session` 登录。

\* 开发“虚拟替身”创建接口：房主可直接在 `room\_players` 插入数据而无需真实用户授权。





\* \*\*Step 3: 房间生命周期管理\*\*

\* 实现建局、扫码加入、位置分配 API。







\### \*\*第二阶段：核心记分引擎与 WebSocket (Day 4-7)\*\*



\* \*\*Step 4: WebSocket 网关开发\*\*

\* 基于 Socket.io 建立房间频道（Room Channel）。

\* 实现“加入房间”事件，订阅分数变动推送。





\* \*\*Step 5: 记分核心逻辑实现\*\*

\* \*\*1对1结算\*\*：原子化更新 `room\_players.current\_score` 并记录 `score\_logs`。

\* \*\*1对多（全收）\*\*：后端执行 $1 \\times 3$ 循环计算，确保事务一致性。





\* \*\*Step 6: 防呆撤销系统\*\*

\* 根据 `room\_id` 查询最新的有效流水，执行反向扣分操作，并标记该 log 为 `is\_revoked`。







\### \*\*第三阶段：称号触发引擎与海报数据 (Day 8-10)\*\*



\* \*\*Step 7: 称号规则处理器 (Title Engine)\*\*

\* 编写 `SettlementService`：当房间状态转为 `finished` 时，从 `score\_logs` 拉取全量数据。

\* 按 PRD 优先级逻辑（绝对统治者 > 终极慈善家 > ...）进行数据扫描。





\* \*\*Step 8: 海报数据接口\*\*

\* 聚合最终得分、匹配称号文案及视觉特效标识，下发给前端 Canvas 渲染。







\### \*\*第四阶段：性能调优与稳定性测试 (Day 11-14)\*\*



\* \*\*Step 9: Redis 缓存介入\*\*

\* 将热点房间的分数同步写入 Redis，异步刷入 PostgreSQL，提升响应速度。





\* \*\*Step 10: 极端场景测试\*\*

\* 压力测试：模拟 4 人同时疯狂点击 `+50` 筹码，验证 WebSocket 推送是否存在延迟或数据乱序。

\* 边界测试：连续撤销直到初始状态、全虚拟人房间结算等。

\### 专家建议：



关于 PRD 中提到的\*\*“第一人称隐形锚点交互”\*\*，后端在处理请求时，\*\*必须\*\*校验 `operator\_id` 的权限。如果是房主在替别人记分，API 需要明确传入 `from\_player\_id` 和 `to\_player\_id`，防止数据串线。
