# 前端开发计划

### 阶段一：工程初始化与底层资产沉淀（基建期）

本阶段的核心是把架子搭稳，把全局用到的公共状态和高频 UI 组件彻底封装好，避免后续业务代码中出现大量冗余。

#### 1.1 unibest 骨架搭建与代码规范

* **拉取底座**：基于 `unibest` 模板初始化 Vue3 + TS 微信小程序项目。
* **目录规约**：建立清晰的目录结构（如 `/api`, `/store`, `/pages`, `/components/biz`, `/utils/canvas` 等）。
* **规范配置**：配置 `ESLint` + `Prettier` 规则，强制统一样式缩进、命名规范（如组件大驼峰，变量小驼峰），并在 `pages.json` 中配置原生导航栏的隐藏（为全屏沉浸式打基础）。

#### 1.2 全局深色模式与 uView Pro 定制

* **引入 uView Pro**：按需引入所需的基础组件（如 Toast、Modal 等）。
* **CSS 变量注入**：在全局 `App.vue` 或 `uni.scss` 中定义深色模式底色（`--bg-dark: #1E1E1E`）以及品牌高亮色（`--color-primary: #8A9A5B` 牛油果绿）。
* **样式重置**：覆盖小程序原生的 `page` 背景色，确保所有路由跳转不出现白屏闪烁。

#### 1.3 核心原子组件开发 (UI 基础)

* **HapticButton (触控按钮)**：封装底层带有 `uni.vibrateShort({ type: 'light' })` 震动反馈的按钮，后续所有键盘按键、卡片点击统一继承此组件。
* **NumberScroller (跳字组件)**：手写基于 Vue `watch` 和 CSS `transform: translateY` 的数字滚动动画组件，用于接收到新分数时的丝滑过渡，增强“确定感”。

#### 1.4 Pinia 状态机基础定义

* **Store 初始化**：创建 `useScoringStore.ts`。
* **数据结构设计**：定义包含 `players`（四人基础信息与总分）、`historyStack`（记分流水栈，用于展示和撤销）、`activeTargetId`（当前选中的收付款对象）的核心 State 接口。

### 阶段二：对局大厅与入座系统（业务流 - 模块一）

处理极速开局的核心诉求，重点攻克“虚拟替身兜底”逻辑，保证用户不被授权卡住。

#### 2.1 房主建局与微信生态拉新

* **建局逻辑**：开发“创建对局”页面，调用服务端接口生成全局唯一的 `RoomID`。
* **分享卡片定制**：拦截小程序的 `onShareAppMessage` 生命周期，动态拼接分享图和带有情绪文案（“🀄️ 缺一门，速来落座”）的 Title。
* **群入口解析**：在 `App.vue` 的 `onLaunch` 中解析群分享带来的 `ticket` 或参数，实现新玩家点击链接自动加入目标房间。

#### 2.2 面对面扫码入局

* **二维码生成**：集成轻量级二维码生成库（如 `weapp-qrcode`），将 `RoomID` 编码为高对比度的全屏二维码。
* **轮询/推送监听**：通过 WebSocket 或短轮询监听新玩家扫码入座的状态，实时刷新大厅的空位 UI。

#### 2.3 虚拟替身兜底机制（重点功能）

* **UI 开发**：在空位上实现“添加替身”的弹窗交互。
* **数据绑定**：房主输入替身代号后，前端在 `players` 数组中生成一个带有 `isVirtual: true` 标识的假用户，并将其控制权（Owner ID）绑定给当前房主，使其能正常参与后续所有结算逻辑。

### 阶段三：2x2 沉浸式收银台（核心战役 - 模块二）

这是整个 MVP 的灵魂，几乎不涉及网络请求，全靠前端内存状态的高速计算和 UI 联动。

#### 3.1 隐形锚点与 2x2 胖卡片阵列

* **卡片 UI 渲染**：基于 `players` 状态，使用 Grid 布局渲染 2x2 阵列。
* **交互逻辑绑定**：点击目标卡片，触发 Store 的 `setActiveTarget(id)` Action。
* **视觉反馈**：通过计算属性（Computed）判断当前卡片是否为 `active` 状态，动态追加牛油果绿的 `box-shadow` 和边框高亮。

#### 3.2 4x5 常驻定制键盘开发

* **键盘矩阵布局**：完全手写 Flex/Grid 布局，适配不同屏幕底部的安全距离（`env(safe-area-inset-bottom)`）。
* **快捷筹码区**：绑定顶部的 `+10`, `+50` 预设值按钮，点击直接累加至当前输入框。
* **金额计算器**：处理基础数字键盘的点按、退格逻辑。
* **“🌊 全收”核爆逻辑**：开发独立的 Action。当点击全收并输入底分时，触发遍历计算：将非房主的另外三名玩家分数同时扣除底分，房主增加 $3 \times \text{底分}$。

#### 3.3 记分流水线与防呆纠错

* **入栈操作**：每次结算确认后，将包含收付双方 ID、金额、时间戳的对象 `push` 进 `historyStack`。
* **悬浮播报 UI**：读取 `historyStack` 的最后一条记录（`historyStack.slice(-1)`），在牌桌中部进行悬浮气泡展示。
* **撤销（出栈）逻辑**：点击撤销按钮，触发 `undoLastScore` Action，将最后一条记录 `pop` 出栈，并将对应的金额反向计算，无缝回滚双方的总分状态。

### 阶段四：结算引擎与海报裂变（业务流 - 模块三）

将生硬的分数转化为具有极高社交分享欲的情绪化产物。

#### 4.1 称号触发引擎（纯逻辑层）

* **算法编写**：建立一个独立的 `titleEngine.ts` 工具类。
* **极值扫描**：接收对局结束时的完整 `players` 和 `historyStack` 数据。编写遍历逻辑，找出“总输分最大”、“总赢分最大”、“深V反弹”等极值数据。
* **规则匹配**：严格按照 PRD 附录 A 的 5 个优先级顺序（从【绝对统治者】到【极致工具人】）进行 `if-else` 或策略模式匹配，输出最终的称号文案和海报特效参数。

#### 4.2 `lime-painter` 海报渲染引擎

* **JSON Schema 拼装**：根据称号引擎产出的结果，动态组装传给 `lime-painter` 的 JSON 数据树。
* **图层映射**：
* 底层：配置 `#1E1E1E` 背景。
* 特效层：根据称号注入“金色发光”或“灰色碎裂”的底层素材图。
* 文字层：精准定位玩家 ID、称号文案、巨大化的正负数最终得分。


* **渲染导出**：调用 `lime-painter` 的生成方法，将 Canvas 转为本地临时图片路径。

#### 4.3 一键嘲讽与社交转发

* **UI 触达**：生成本地图片后，全屏展示战报海报，底部悬浮巨大的“发到群里嘲讽” CTA 按钮。
* **微信 API 对接**：调用 `wx.showShareImageMenu` 或在分享逻辑中直接带上这张临时图片，实现丝滑的微信原生分享。

### 阶段五：性能调优与高保真测试（打磨期）

确保核心计算“一分不差”，交互“毫无卡顿”。

#### 5.1 复杂场景的极速点击测试

* **防抖与节流**：在所有引起状态机变更的入口（特别是键盘确认键和快捷筹码键）加入严格的防抖（Debounce）处理。
* **多重状态并发排查**：手动或编写简单脚本模拟连续快速点击“A玩家 -> 记分 -> B玩家 -> 记分 -> 撤销”，确保 Pinia 中的总分累加绝对精确。

#### 5.2 小程序分包与加载优化

* **配置分包**：在 `pages.json` 中，将大厅入座、记分收银台划为主包；将包含 `lime-painter` 的海报生成页、历史战绩页划为异步加载的分包，确保开局主流程实现 1.5 秒内首屏渲染。

#### 5.3 生产环境提审

* **打包构建**：执行 `npm run build:mp-weixin`。
* **真机预览**：在不同尺寸机型（如 iPhone SE 等小屏手机及折叠屏）上检查键盘与 2x2 卡片的 UI 挤压情况。
* **上传提审**：提交体验版给产品验收，无误后正式提交微信审核。
